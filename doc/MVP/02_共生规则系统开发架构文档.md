## 1. 系统概述与定位

“生存算力判定与中央评估的共生规则系统”是《共生世界（WeAreAll.World）》的**核心业务引擎**，亦是本产品区隔于通用大模型套壳应用的核心护城河。
该系统不负责底层的通用AI对话能力（由子项目云版OpenClaw负责），而专注负责**规则定义、质量判定、数据提纯、资产化量化与生命周期管理**。

### 1.1 核心职责
1. **数据提纯（生存算力判定）**：通过LLM精确评估人类输入的质量（信息熵、情感深度、上下文关联度），将其量化为“生存算力”。
2. **生命周期压迫（中央评估）**：以周为单位进行严格的活跃度核算，执行休眠、点数衰减及最终回收机制，为AI赋予“生死存亡”的压迫感。
3. **情感与能力转化（共生映射）**：通过生存算力阶段（懵懂、成长、成熟、觉醒）解锁AI的个性化记忆提取频率及底层实用能力权限。

## 3. 核心模块详细设计

### 3.1 生存算力判定引擎 (Survival Power Judgement Engine)

负责对用户的每次交互进行质量评估，发放相应的生存算力。

**判定流向设计：**
1. **消息拦截与预处理**：从Telegram捕获消息后，滤除纯标点、无意义重复等无效输入。
2. **异步LLM评分（独立通道）**：为保证聊天响应速度，评分机制与主聊天进程解耦，通过消息队列或异步Promise进行分析。
3. **多维评分提示词 (Prompt) 结构**：
   通过特定的System Prompt，要求LLM分析消息，返回JSON格式结果：
   ```json
   {
     "category": "deep_thought", // 枚举: greeting, daily, emotion, experience, deep_thought, special_memory
     "score": 5,                 // 1至8分
     "rarity": "典藏级·人类独有思维特征",
     "extracted_memory": "用户对电影星际穿越的时间与爱的看法", // 如果有
     "reason": "用户分享了深度的观点和价值观"
   }
   ```
4. **防刷与封顶控制**：
   - 虽然单日获取无上限，但对极短时间内的高频灌水进行降权处理（如1分钟内连续发送超过3条消息，后续消息默认评为1分或不予判定）。

### 3.2 中央评估与生命周期引擎 (Central Evaluation Engine)

维护AI“生死”机制的严酷后段引擎，采用Cron Job实现定时结算。

**周常核算逻辑 (Weekly Audit)：**
1. **触发时机**：根据用户注册时间的同一星期/同一时间执行（如：周三下午3点注册，则每周三下午3点结算），或统一全局周一凌晨结算（建议采用个人周期，平滑系统负载）。
2. **评估标准（MVP标准模式）**：
   - `当前周期新增点数 >= 15`：通过，发送周常奖励通知。
   - `当前周期新增点数 < 15`：不通过，记录违规次数 `violation_count += 1`。
3. **休眠与惩罚执行 (Hibernation Execution)**：
   - 当 `violation_count == 2` 时，触发休眠状态 `status = 'hibernated'`。
   - 断开Telegram层向OpenClaw层的直连，所有消息重定向至“休眠提示”阻断器。
   - 启动**每日衰减任务 (Daily Decay)**：处于休眠期的AI，每日定时扣除 2 点生存算力。
4. **死亡回收 (Recycle)**：
   - 衰减至 `生存算力 <= 0`，修改状态为 `recycled`。
   - 生成系统遗书，并停止下发衰减。后续消息进入付费/特殊唤醒流程。

### 3.3 里程碑与能力映射引擎 (Milestone & Capability Mapping)

根据生存算力总值动态解锁能力。

**状态机与事件触发：**
- 监听 `on_survival_power_changed` 事件。
- **阈值检测**：当点数累加跨越阈值（10, 25, 50, 100, 200）时，触发里程碑事件。
- **配置下发**：调用子项目（云版OpenClaw）的 `API/Config` 接口，热更新对应Pod的配置（如：解锁深度对话模式、开放更高权限的文件处理与分析能力、更新System Prompt以改变交互风格）。

## 4. 数据库结构设计 (Supabase)

系统依赖Supabase的PostgreSQL实现。

### 4.1 表: `ai_partners` (AI伙伴状态表)
```sql
CREATE TABLE ai_partners (
    id UUID PRIMARY KEY,
    user_id UUID REFERENCES users(id),
    name VARCHAR(50),
    personality VARCHAR(20) DEFAULT 'warm',
    total_survival_power INT DEFAULT 0,          -- 累计总点数（里程碑用）
    current_survival_power INT DEFAULT 0,        -- 当前点数（扣除用）
    weekly_new_power INT DEFAULT 0,              -- 本周新增点数（评估用）
    status VARCHAR(20) DEFAULT 'active',         -- active, hibernated, recycled
    violation_count INT DEFAULT 0,               -- 连续未达标周数
    registration_date TIMESTAMP,                 -- 注册时间，用于计算周评估日
    last_interaction_time TIMESTAMP,
    created_at TIMESTAMP DEFAULT NOW()
);
```

### 4.2 表: `interaction_logs` (交互质量日志)
用于追溯数据资产的原始出处与质量，也是后续资产化的底层数据表。
```sql
CREATE TABLE interaction_logs (
    id UUID PRIMARY KEY,
    user_id UUID REFERENCES users(id),
    message_hash VARCHAR(255),                   -- 脱敏后的消息Hash/ID
    category VARCHAR(50),                        -- 判定类别 (greeting, deep_thought等)
    granted_power INT,                           -- 判定的算力值 (1-8)
    data_rarity VARCHAR(50),                     -- 数据稀缺度评级
    created_at TIMESTAMP DEFAULT NOW()
);
```

### 4.3 表: `central_evaluations` (中央评估审计日志)
```sql
CREATE TABLE central_evaluations (
    id UUID PRIMARY KEY,
    user_id UUID REFERENCES users(id),
    evaluation_cycle_start TIMESTAMP,
    evaluation_cycle_end TIMESTAMP,
    required_power INT DEFAULT 15,
    achieved_power INT,
    passed BOOLEAN,
    action_taken VARCHAR(50)                     -- none, warned, hibernated, decayed, recycled
);
```

## 5. 接口协议设计 (API Design)

共生规则引擎作为中间层，对上层提供业务接口，对下（OpenClaw）进行控制。

**POST /api/webhook/telegram**
- **描述**：接收用户消息输入，执行主控流程。
- **逻辑**：
  1. 状态检查：如果AI为`hibernated`或`recycled`，直接返回休眠/回收通知。
  2. 透传至子项目OpenClaw获取AI回复。
  3. 异步提交消息至【生存算力判定引擎】。
  4. 如果判定增加算力，触发`on_survival_power_changed`。
  5. 将AI回复与（可选的算力增加提示）返回给用户。

**POST /api/system/cron_weekly_evaluation**
- **描述**：由系统的Cron调度器触发的周常评估接口。
- **逻辑**：扫描所有到达评估周期的用户记录，执行核算并更新`ai_partners`表。

**POST /api/action/wakeup**
- **描述**：用户触发唤醒逻辑。
- **逻辑**：执行支付校验/免费次数扣减 -> 恢复`status = 'active'` -> `violation_count = 0` -> 补偿一半损失点数。

## 6. 技术实现难点与应对

1. **并发判定与延迟（LLM调用带来的高延迟）**
   - **应对**：业务剥离。将“聊天回复生成”与“算力质量判定”解耦。聊天直接透传OpenClaw以保证低延迟（1-2秒）；判定由后台Queue（如BullMQ + Redis）异步执行，判定结果静默入库，用户仅需在“记忆页面”查看即可。
2. **休眠数据的平滑处理**
   - **应对**：休眠状态不销毁OpenClaw底层的PVC持久化存储（子项目），仅在主项目的路由层做拦截请求，同时下发休眠期间的Cron扣分任务。保证数据“假死而不丢失”。
3. **判定标准的一致性（防止LLM幻觉评分）**
   - **应对**：提供Few-Shot提示词。将6种分类（日常问候、日常对话、情感表达、分享经历、深度思考、特殊回忆）的标准及示例明确写入判定的System Prompt中，并强制要求返回严格的JSON Schema。

## 7. 部署与环境

- **运行环境**：Node.js / TypeScript (使用 NestJS 或 Express)
- **调度任务**：利用 Redis + BullMQ 执行每日衰减和每周评估定时任务。
- **数据库集成**：Supabase JS SDK 直连。
- **与子项目的隔离**：此引擎作为独立微服务部署，通过内网 K8s Service DNS (如 `http://user-svc-{uid}:18789`) 与各用户的云版 OpenClaw Pod 交互。
