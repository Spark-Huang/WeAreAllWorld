## 1. 系统概述与定位

“生存算力判定与中央评估的共生规则系统”是《共生世界（WeAreAll.World）》的主项目**核心业务引擎**，亦是本产品区隔于通用大模型套壳应用的核心护城河。
该系统不负责底层的通用AI对话、任务执行与记忆持久化能力（这些由子项目：云版OpenClaw负责），而专注负责**规则定义、质量判定、数据提纯、资产化量化与生命周期管理**。

### 1.1 核心职责与商业价值
1. **数据引擎提纯（生存算力判定）**：通过LLM精确评估人类输入的质量（信息熵、情感深度、上下文关联度），将其量化为“生存算力”。这是获取AGI时代最稀缺的差异化人类私域数据的核心筛滤器。
2. **生命周期压迫（中央评估）**：以周为单位进行严格的活跃度核算，执行休眠、算力衰减及最终回收机制。为AI赋予“生死存亡”的压迫感，极大提升用户留存与数据贡献动力。
3. **情感与实用能力映射（共生飞轮）**：通过生存算力里程碑（初识、相知、默契、灵魂伴侣），动态解锁AI的个性化记忆提取频率及底层实用能力权限，将情感养成直接转化为10倍效能的工具优势。

## 3. 核心模块详细设计

### 3.1 生存算力判定引擎 (Survival Power Judgement Engine)

负责对用户的每次交互进行质量评估，发放相应的生存算力。这是高质量数据收集的关键闸口。

**判定流向设计：**
1. **消息拦截与预处理**：从Telegram捕获消息后，滤除纯标点、无意义重复等无效输入。
2. **异步评分通道解耦**：为保证聊天响应速度，聊天请求直接透传给子项目OpenClaw生成回复。质量评分机制与主聊天进程完全解耦，通过消息队列（如BullMQ）异步分析。
3. **多维评分提示词 (Prompt) 结构**：
   专门的评估LLM实例负责分析消息，基于《MVP需求文档》定义的6大层级，返回JSON格式结果：
   ```json
   {
     "category": "deep_thought", // 枚举: greeting, daily, emotion, experience, deep_thought, special_memory
     "score": 5,                 // 1至8分
     "rarity": "典藏级·人类独有思维特征",
     "extracted_memory": "用户对电影星际穿越的时间与爱的看法", // 提取关键语义用于资产化
     "reason": "用户分享了深度的观点和价值观"
   }
   ```
4. **防刷与封顶控制**：
   - 每日获取总点数**无上限**。
   - 对极短时间内的高频灌水进行降权处理（如1分钟内连续发送超过3条短消息，后续消息默认评为1分或暂不触发判定）。
   - 包含主动引导提示词生成逻辑（每1-2天触发一次），引导用户产出高价值数据。

### 3.2 中央评估与生命周期引擎 (Central Evaluation Engine)

维护AI“生死”机制的严酷后段引擎，制造强粘性沉没成本。

**生命周期核算逻辑：**
1. **周常核算 (Weekly Audit)**：
   - 基于用户注册时间按周执行Cron Job。
   - **判定标准**：`本周新增生存算力 >= 15点` 则通过；否则记录违规 `violation_count += 1`。
2. **休眠与惩罚执行 (Hibernation Execution)**：
   - 连续两周不通过（`violation_count == 2`），触发休眠状态 `status = 'hibernated'`。
   - **业务阻断**：休眠期间，主项目切断Telegram向OpenClaw实例的路由转发，全部拦截并返回“AI已休眠，请唤醒”的系统提示。
   - **算力衰减 (Daily Decay)**：休眠期间，每日定时任务扣除 2 点生存算力。
3. **唤醒与回收 (Wakeup & Recycle)**：
   - 若算力降至 `<= 0`，修改状态为 `recycled`，后续必须通过付费或其他高门槛途径复活。
   - 成功唤醒后，返还休眠期间损失的 50% 算力，并重置 `violation_count = 0`。

### 3.3 里程碑与能力映射引擎 (Milestone & Capability Mapping)

将情感投入转化为实用工具价值的核心转化器。

**能力映射流转：**
1. **事件监听**：监听 `on_survival_power_changed` 事件，实时核算总点数（`total_survival_power`）。
2. **阈值晋升**：当总点数跨越 10, 25, 50, 100, 200 等节点时，触发相应里程碑事件。
3. **子项目上下文热更新**：
   - 动态更新下发给子项目OpenClaw的 `UserContext`。
   - **实用能力增强系数 (practicalAbilityBoost)**：随着点数增加，授权OpenClaw调用更高级的底层Skill（如复杂文件解析、多步骤代码运行）。
   - **上下文注入**：通知OpenClaw解锁“专属记忆主动提取”频率、改变交互风格（Style Adaptation）配置等。

## 4. 数据库结构设计 (Supabase)

系统依赖Supabase的PostgreSQL实现。设计必须支持未来数据的脱敏与资产化提取。

### 4.1 表: `users` (用户基础表)
*(标准Supabase Auth扩展)*
包含基础信息与注册渠道追踪。

### 4.2 表: `ai_partners` (AI伙伴状态表)
```sql
CREATE TABLE ai_partners (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    user_id UUID REFERENCES users(id) ON DELETE CASCADE,
    name VARCHAR(50),
    personality VARCHAR(20) DEFAULT 'warm',
    growth_stage VARCHAR(20) DEFAULT '懵懂期',     -- 根据总点数映射的阶段名称
    total_survival_power INT DEFAULT 0,          -- 累计总点数（只增不减，用于里程碑）
    current_survival_power INT DEFAULT 0,        -- 当前点数（休眠会扣除）
    weekly_new_power INT DEFAULT 0,              -- 本周新增点数（评估核算用）
    status VARCHAR(20) DEFAULT 'active',         -- active, hibernated, recycled
    violation_count INT DEFAULT 0,               -- 连续未达标周数
    registration_date TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    last_interaction_time TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);
```

### 4.3 表: `interaction_logs` (交互质量与数据资产池)
大模型训练的绝佳私域数据集。设计支持严格脱敏。
```sql
CREATE TABLE interaction_logs (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    user_id UUID REFERENCES users(id),
    message_hash VARCHAR(255),                   -- 加密/脱敏后的消息摘要标识
    category VARCHAR(50),                        -- 判定类别 (greeting, deep_thought等)
    granted_power INT,                           -- 判定的算力值 (1-8)
    data_rarity VARCHAR(50),                     -- 数据稀缺度评级
    ai_understanding JSONB,                      -- 存储LLM判定的详细JSON结果(情感/提纯记忆)
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);
```

### 4.4 表: `central_evaluations` (中央评估审计流水)
记录每一次周常评估与生死判决。
```sql
CREATE TABLE central_evaluations (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    user_id UUID REFERENCES users(id),
    evaluation_cycle_start TIMESTAMP WITH TIME ZONE,
    evaluation_cycle_end TIMESTAMP WITH TIME ZONE,
    required_power INT DEFAULT 15,
    achieved_power INT,
    passed BOOLEAN,
    action_taken VARCHAR(50),                    -- none, warned, hibernated, decayed, recycled
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);
```

## 5. 接口协议设计 (API Design)

共生规则引擎作为中间层，对上层接入客户端（Telegram），对下控制OpenClaw沙箱。

### 5.1 Telegram Webhook 核心路由
**`POST /api/webhook/telegram`**
- **逻辑流**：
  1. 鉴权与用户映射。
  2. 状态检查：如果AI为 `hibernated`，拦截并返回系统休眠通知，阻断请求下发。
  3. 透传逻辑：向子项目OpenClaw的独立Pod (e.g., `http://user-svc-{uid}:18789/api/chat`) 发送带 `UserContext` 头的对话请求。
  4. 旁路判定：异步提交消息文本至【生存算力判定引擎】队列。
  5. 返回响应：将OpenClaw的生成内容返回给Telegram。

### 5.2 系统内部 Cron API
**`POST /api/system/cron/weekly_evaluation`**
- **逻辑**：扫描 `registration_date` 对齐到今日当前小时的所有活跃记录，执行周常核算，重置 `weekly_new_power`，根据结果更新 `status` 与 `violation_count`。

**`POST /api/system/cron/daily_decay`**
- **逻辑**：扫描所有 `status = 'hibernated'` 且 `current_survival_power > 0` 的记录，执行 `-2` 点数衰减。

### 5.3 业务交互 API
**`POST /api/action/wakeup`**
- **逻辑**：用户触发唤醒。处理免费次数或支付 -> 恢复 `status = 'active'` -> `violation_count = 0` -> 算力补偿 -> 通知用户。

## 6. 技术实现难点与应对

1. **LLM判定带来的高延迟与成本激增**
   - **痛点**：若在对话主链路同步等待质量判定，会导致回包极慢，且每次对话调用2次LLM成本过高。
   - **应对**：
     - **异步解耦**：算力判定通过BullMQ后台静默执行，结果推入数据库，用户后续通过“查看记忆”面板感知。
     - **批量聚合评估 (Batch Processing)**：对于高频零碎对话，可利用滑动窗口聚合N条消息，进行一次集中式上下文质量评估，大幅降低判定Token消耗。

2. **主子项目的数据同步一致性**
   - **痛点**：主项目维护全局生命周期（算力、阶段），子项目维护具体对话历史与技能记忆。
   - **应对**：采用**Context Injection（上下文注入）**机制。主项目在每次转发聊天请求给OpenClaw时，在HTTP Header `X-User-Context` 中携带最新的算力等级、阶段标识及增强系数，子项目无需反向查询主数据库。

3. **休眠数据的平滑处理**
   - **痛点**：休眠状态如何处理子项目资源。
   - **应对**：由于子项目采用CCI Serverless竞价实例，长时间无请求的Pod可由K8s策略主动缩容/回收，但底层PVC持久卷保持不销毁。唤醒时重新拉起Pod挂载原PVC，实现业务上的“假死而数据不丢失”。

## 7. 部署与技术栈推荐

- **核心运行环境**：Node.js / TypeScript (推荐 NestJS 架构以应对复杂的依赖注入和定时任务管理)。
- **队列与调度**：Redis + BullMQ，用于处理异步算力判定队列和Cron定时任务。
- **数据库**：Supabase (PostgreSQL) + Prisma/TypeORM，直连以利用行级安全和实时订阅特性。
- **子项目协同**：部署在与华为云CCI处于同一VPC内的主控节点上，通过内网 K8s Service DNS 高速访问各用户的云版 OpenClaw Pod 集群。
