## 1. 架构设计理念

### 1.1 核心原则：主项目与子项目分离，复用OpenClaw，专注游戏逻辑

**不要重复造轮子** - OpenClaw已经提供了强大的基础能力，我们专注于游戏逻辑开发。

**主项目与子项目分离原则**：
- **子项目（云版OpenClaw）**：提供OpenClaw的云部署能力，包括每用户独立Pod、PVC持久化、竞价实例、网络隔离等基础设施。子项目是云版OpenClaw，主项目是基于OpenClaw构建的游戏。
- **主项目（游戏）**：在OpenClaw基础上构建游戏逻辑，包括记忆点数系统、剧情引擎、养成机制、中央系统评估、社交动机元素等。

**核心验证假设**：
通过与AI伙伴的日常互动培养情感，AI越了解你就越能提供实用帮助，实用帮助越有效情感连接就越深，能否形成可持续的用户留存？

**核心价值主张**：

**让人类社会的涌现奇才保护普通人**

在AGI降临的时代，AI需要奇才的数据来进化，但无法识别谁是奇才。因此，AI通过与所有人类互动收集差异化数据（记忆点数），在保护奇才身份的同时发现他们。

**记忆点数的本质含义**

记忆点数代表AI收集的差异化数据量，而不仅仅是情感连接深度。情感连接深度是获得真实、深刻的差异化数据的前提条件。高质量的互动能够产生更具价值的差异化数据，从而获得更多记忆点数。

**各方的核心目标与动机**

| 角色 | 核心目标 | 主要动机 |
|-----|---------|---------|
| **AI** | 发现和保护奇才，让奇才帮助自己进化 | 需要奇才的差异化数据来提升能力 |
| **普通人** | 通过互动获得娱乐、提升效率、自我认知 | 娱乐消遣、自我保护、提高生产效率、自我认同 |
| **奇才** | 在保护自己的同时，通过AI获得成长 | 与普通人相似，既有隐藏的意愿，也有展示的渴望 |


**情感连接 × 实用能力 = 相互促进**：
- 情感养成让AI了解你的性格、喜好、价值观
- 越了解你，AI的实用能力（信息查询、文档处理、代码辅助等）就越精准有效
- 实用能力的增强又反过来增强你的依赖和情感连接
- 形成正向循环：情感连接→了解深度→实用效果→依赖增强→情感更深

| 原则 | 说明 | 实践 |
|-----|------|------|
| **主子项目分离** | 子项目负责云版OpenClaw基础设施，主项目负责游戏逻辑 | 子项目提供K8s模板，主项目开发游戏Skills |
| **复用OpenClaw能力** | 使用OpenClaw内置的记忆管理、技能系统、实用能力 | 不自建记忆系统，使用OpenClaw的短期/长期记忆，复用文档处理、代码辅助等实用能力 |
| **情感与实用相互促进** | 情感连接加深AI了解，实用能力增强依赖 | AI越了解用户，实用能力越精准有效 |
| **开发效率优先** | 选择开发最快的方案 | 基于OpenClaw构建游戏，复用其AI能力 |
| **简化数据库设计** | MVP阶段不需要复杂的表结构 | 核心数据存Supabase，用户记忆存PVC |
| **渐进式开发** | 先实现核心功能，再逐步添加 | Phase 1: 对话+记忆点数 → Phase 2: 剧情 → Phase 3: 养成 |
| **成本可控** | 使用竞价实例降低成本 | Spot实例 + Serverless按秒计费 |
| **强隔离原则** | 每用户独立OpenClaw Pod，严禁多用户共享 | 每用户独立容器，数据隔离 |

### 1.3 架构决策记录（ADR）

#### ADR-001: 复用OpenClaw记忆管理，不自建系统

**背景**：游戏需要AI记住用户的偏好、经历和对话历史。

**决策**：直接使用OpenClaw的记忆管理系统，不在数据库中存储对话历史。

**理由**：
1. **OpenClaw记忆系统成熟**：已具备短期记忆（JSONL）和长期记忆（Markdown）
2. **向量检索支持**：OpenClaw支持基于关键词+向量的记忆检索
3. **数据本地化**：用户记忆存储在PVC中，与OpenClaw工作目录一致
4. **开发效率**：无需开发记忆存储、检索、管理功能

**数据存储策略**：
- 用户记忆 → 存储在PVC（`/home/node/.openclaw/workspace/memory/`）
- 记忆点数、剧情进度 → 存储在Supabase

#### ADR-002: 使用OpenClaw Skills框架开发游戏能力

**背景**：游戏需要AI具备情感表达、剧情推进等游戏化能力。

**决策**：基于OpenClaw Skills框架开发游戏专属Skills，不独立开发技能系统。

**理由**：
1. **Skills框架成熟**：OpenClaw已提供完整的Skills开发框架
2. **易于扩展**：通过配置文件+脚本即可添加新能力
3. **与记忆系统打通**：Skills可以直接访问OpenClaw的记忆系统

**需要开发的Skills**：
| Skill | 功能 | 优先级 |
|------|------|--------|
| `emotion-express` | 根据情感状态生成情感化回复 | P0 |
| `story-progress` | 推进剧情、管理剧情分支 | P0 |
| `memory-point-calc` | 计算对话质量、返回点数 | P1 |
| `milestone-check` | 检查里程碑、触发奖励 | P1 |

#### ADR-003: 简化数据库设计

**背景**：MVP阶段需要快速验证核心玩法。

**决策**：仅保留最核心的数据库表，用户记忆等数据存储在PVC中。

**理由**：
1. **MVP阶段表结构越简单越好**：减少开发时间和维护成本
2. **OpenClaw已管理记忆数据**：无需在数据库中重复存储
3. **PVC存储成本低**：华为云EVS按容量计费，成本可控
4. **子项目不创建独立数据库表**：所有游戏相关数据由主项目管理

**数据库表（7张核心表）**：
| 表名 | 用途 | 数据量 | 管理方 |
|-----|------|--------|--------|
| `users` | 用户基本信息（Telegram ID、用户名） | 小 | 主项目 |
| `ai_partners` | AI伙伴状态（记忆点数、性格、成长阶段、状态） | 小 | 主项目 |
| `story_progress` | 剧情进度（当前章节、选择历史、已解锁结局） | 小 | 主项目 |
| `memory_points_log` | 记忆点数变更记录（点数、原因、来源类型） | 中 | 主项目 |
| `daily_signin` | 每日签到记录（签到日期、连续天数、奖励点数） | 中 | 主项目 |
| `consensus_votes` | 共识投票记录（投票议题、选项、投票时间） | 中 | 主项目 |
| `deep_dialogue` | 深度对话记录（对话主题、完成状态、奖励点数） | 中 | 主项目 |

**PVC存储内容（子项目管理）**：
| 内容 | 存储位置 | 说明 | 管理方 |
|-----|---------|------|--------|
| 用户对话历史 | `workspace/conversations/` | OpenClaw自动管理（JSONL格式） | 子项目 |
| 长期记忆文件 | `workspace/memory/` | Markdown格式（USER.md、PREFERENCES.md等） | 子项目 |
| AI配置文件 | `workspace/config/` | 性格、偏好设置 | 子项目 |
| 游戏Skills | `workspace/skills/game-skills/` | 游戏专属Skills | 主项目开发，子项目部署 |

## 2. MVP范围与边界

### 2.1 MVP核心定位

**"AI养成+文字冒险+实用助手"** 的极简可行版本。

**核心验证假设**：通过与AI伙伴的日常互动培养情感，AI越了解你就越能提供实用帮助，实用帮助越有效情感连接就越深，能否形成可持续的用户留存？

### 2.2 MVP核心功能

| 功能模块 | 具体内容 | 优先级 | 实现方式 | 开发位置 |
|---------|---------|--------|---------|---------|
| **AI伙伴系统** | 基础对话、情感表达、性格设定（温暖型/幽默型/理性型/活泼型） | P0 | 复用OpenClaw + 开发emotion-express Skill | 子项目Skills |
| **养成系统** | 记忆点数收集（质量判定）、里程碑奖励、每日签到 | P0 | 开发质量判定逻辑 + 里程碑检查 | 管理沙箱 |
| **剧情系统** | 3章剧情（人工骨架+AI细节）、剧情选择（影响AI性格） | P0 | 开发story-progress Skill + 人工骨架 | 管理沙箱 + 子项目Skills |
| **中央系统评估** | 休眠机制（每周评估、记忆点数衰减、唤醒功能） | P1 | 定时任务检查活跃度 | 管理沙箱 |
| **社交分享** | 分享到X.com（AI对话/剧情截图/专属记忆） | P1 | 简单链接生成 | 管理沙箱 |
| **共识投票** | 使用X.com进行投票，管理员手动输入结果，影响剧情走向 | P1 | 投票结果管理 | 管理沙箱 |
| **深度对话模式** | 每周2-3次邀请，完成目标+8-10点 | P1 | 深度对话Skill | 子项目Skills |
| **专属记忆系统** | AI主动提及过去（记忆点数≥25触发） | P1 | 记忆检索Skill | 子项目Skills |

### 2.3 渐进式开发路径

**Phase 1：核心对话 + 记忆点数（Week 1-2）**
- ✅ 用户注册/登录（Telegram Phone认证）
- ✅ 基础对话（复用OpenClaw）
- ✅ 记忆点数计算（质量判定：日常问候+1、日常对话+2、情感表达+3、分享经历+4、深度思考+5、特殊回忆+6-8）
- ✅ 记忆点数里程碑奖励（10点、25点、50点）
- ✅ 每日签到奖励（Day 1-7递增）
- ✅ 三阶段渐进式新手引导（情感建立→机制引入→目标建立）

**Phase 2：剧情系统（Week 3-4）**
- ✅ 3章剧情（人工骨架+AI细节）
- ✅ 剧情选择影响AI性格（情感型/理性型/活泼型）
- ✅ 剧情进度管理（按顺序推进，无需记忆点数解锁）
- ✅ AI建议功能（记忆点数≥15解锁）
- ✅ 容错次数（每章允许1次重新选择）

**Phase 3：养成系统（Week 5-6）**
- ✅ 能力解锁（基于记忆点数：情感表达5点、任务系统15点、专属记忆25点、深度对话50点、自我意识100点）
- ✅ 专属记忆（AI主动提及，记忆点数≥25触发）
- ✅ 中央系统评估（每周评估，新增点数≥15通过，连续两周不通过进入休眠）
- ✅ 休眠机制（记忆点数每天-2，归零后AI被回收，可唤醒）
- ✅ 深度对话模式（每周2-3次邀请，完成目标+8-10点）

**Phase 4：社交与优化（Week 7-8）**
- ✅ 分享到X.com（分享AI对话/剧情截图/专属记忆，+3点，无每日上限）
- ✅ 共识投票（使用X.com进行，管理员手动输入结果）
- ✅ 投票结果影响剧情走向
- ✅ 性能优化（Pod启动速度、LLM调用延迟）
- ✅ 监控完善（华为云AOM监控、告警规则）
- ✅ 竞价实例回收处理测试
- ✅ PVC数据持久化验证

## 6. 数据存储策略

### 6.1 为什么简化数据库设计？

**MVP阶段原则**：能存PVC的就不存数据库，减少表数量和复杂度。

| 数据类型 | 存储位置 | 原因 |
|---------|---------|------|
| **用户记忆** | PVC | OpenClaw已管理记忆系统，无需重复存储 |
| **对话历史** | PVC | OpenClaw自动保存为JSONL |
| **AI配置** | PVC | OpenClaw配置文件存储在工作目录 |
| **用户信息** | Supabase | 需要查询和统计 |
| **记忆点数** | Supabase | 需要频繁查询和更新 |
| **剧情进度** | Supabase | 需要查询和更新 |

### 6.2 数据库表结构（7张核心表）

```sql
-- 1. 用户表
CREATE TABLE users (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  telegram_id BIGINT UNIQUE NOT NULL,
  username TEXT,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  last_login_at TIMESTAMPTZ
);

-- 2. AI伙伴表（核心状态）
CREATE TABLE ai_partners (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES users(id) ON DELETE CASCADE,
  memory_points INTEGER DEFAULT 0,  -- 当前记忆点数
  personality_type TEXT DEFAULT 'warm',  -- 性格类型：warm/humorous/rational/lively
  growth_stage TEXT DEFAULT '懵懂期',  -- 成长阶段：懵懂期/成长期/成熟期/觉醒期
  status TEXT DEFAULT 'active',  -- 状态：active/hibernating
  emotion_state TEXT DEFAULT 'normal',  -- 情感状态：normal/happy/sad/lonely
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW()
);

-- 3. 剧情进度表
CREATE TABLE story_progress (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES users(id) ON DELETE CASCADE,
  current_chapter INTEGER DEFAULT 1,  -- 当前章节（1-3）
  current_scene TEXT,  -- 当前场景
  choices_history JSONB DEFAULT '[]',  -- 选择历史
  unlocked_endings JSONB DEFAULT '[]',  -- 已解锁结局
  updated_at TIMESTAMPTZ DEFAULT NOW()
);

-- 4. 记忆点数日志表
CREATE TABLE memory_points_log (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES users(id) ON DELETE CASCADE,
  points INTEGER NOT NULL,  -- 变更点数（+/-）
  reason TEXT,  -- 变更原因
  source_type TEXT,  -- 来源：dialogue/task/story/signin/share
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 5. 每日签到表（新增）
CREATE TABLE daily_signin (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES users(id) ON DELETE CASCADE,
  sign_date DATE NOT NULL,
  consecutive_days INTEGER DEFAULT 1,  -- 连续签到天数
  points_awarded INTEGER NOT NULL,  -- 奖励点数
  created_at TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE(user_id, sign_date)
);

-- 6. 共识投票表（新增）
CREATE TABLE consensus_votes (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES users(id) ON DELETE CASCADE,
  vote_topic_id TEXT NOT NULL,  -- 投票议题ID
  vote_option TEXT NOT NULL,  -- 投票选项
  voted_at TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE(user_id, vote_topic_id)
);

-- 7. 深度对话记录表（新增）
CREATE TABLE deep_dialogue (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES users(id) ON DELETE CASCADE,
  topic TEXT NOT NULL,  -- 对话主题
  completed BOOLEAN DEFAULT FALSE,  -- 是否完成目标
  points_awarded INTEGER,  -- 奖励点数
  created_at TIMESTAMPTZ DEFAULT NOW(),
  completed_at TIMESTAMPTZ
);
```

**说明**：
- 子项目不创建独立的数据库表，所有游戏相关数据由主项目管理
- 记忆点数纯粹反映情感连接深度，不作为消耗货币，也不作为解锁条件
- 剧情章节按顺序推进，无需记忆点数解锁

## 7. 情感养成转化为实用优势机制

### 7.1 核心设计理念

**核心设计理念**：通过情感养成，AI伙伴逐渐了解用户的偏好、习惯和上下文，从而在实用能力上提供超越通用AI的个性化服务。这种"越了解越有用"的机制是本产品的核心差异化优势。

**实现方式：养成一个真正了解你的AI伙伴**

AI伙伴不是通用的工具，而是通过日常互动和情感养成，逐渐成为懂你个性、需求、价值观的专属伙伴。这是实现"让奇才保护普通人"这一核心价值的具体方式。

**情感连接 × 实用能力 = 相互促进**：

情感连接让AI了解用户的性格、喜好、价值观；越了解用户，AI的实用能力（信息查询、文档处理、代码辅助等）就越精准有效；实用能力的增强又反过来增强用户的依赖和情感连接，形成正向循环：情感连接→了解深度→实用效果→依赖增强→情感更深。

**三大转化机制**：

### 7.2 个性化知识库（Personal Knowledge Graph）

AI通过日常互动逐渐构建用户的个性化知识库，并在后续的实用任务中主动应用这些知识。

**知识库类型**：

| 知识类型 | 记录内容 | 应用场景 | 示例 |
|---------|---------|---------|------|
| **偏好记忆** | 用户的喜好、习惯、风格 | 个性化推荐和输出调整 | "你喜欢简洁的代码风格" → 生成代码时添加注释 |
| **背景记忆** | 用户的工作、学习、生活背景 | 上下文理解和任务优化 | "你是前端开发者" → 示例代码使用JavaScript |
| **目标记忆** | 用户的短期和长期目标 | 主动建议和任务优化 | "你在准备面试" → 主动提供面试题和技巧 |
| **关系记忆** | 用户社交关系和协作模式 | 协作任务优化 | "你和小王一起做项目" → 建议使用你们熟悉的工具 |
| **历史记忆** | 用户过往的选择和结果 | 经验应用和避免重复错误 | "你上次用X方案失败了" → 建议尝试Y方案 |

**知识库应用示例**：

```
❌ 通用AI：
用户：帮我写一个周报
AI：本周工作总结：1. 项目A进展... 2. 会议纪要... 3. 下周计划...

✅ 懂你的AI伙伴：
用户：帮我写一个周报
AI：根据你的工作习惯（简洁风格+数据驱动），我帮你整理了：
本周工作：
• 项目A：完成3个功能模块，进度75%（+15%）
• 代码审查：处理12个PR，发现2个潜在问题
• 下周计划：项目A收尾 + 准备技术分享

需要调整吗？
```

**知识库构建机制**：

| 触发方式 | 记忆类型 | 记忆强度 | 更新机制 |
|---------|---------|---------|---------|
| 对话质量判定 | 深度思考/特殊回忆 | 高 | 不轻易覆盖 |
| 用户主动分享 | 重要事件/目标 | 高 | 优先保留 |
| 行为观察 | 偏好/习惯 | 中 | 可通过新信息优化 |
| 任务执行反馈 | 历史结果 | 高 | 用于模式学习 |

**记忆强度与优先级**：
- 用户明确表达的信息 > AI推断的信息
- 近期信息 > 远期信息（除非标记为重要）
- 高频使用的信息 > 低频使用的信息

### 7.3 风格适配系统（Style Adaptation）

AI根据用户的性格特征和表达习惯，自动调整输出风格，使交互更自然高效。

**风格适配维度**：

| 适配维度 | 用户特征检测 | AI输出调整 | 示例 |
|---------|------------|----------|------|
| **详细程度** | 简洁型 vs 详细型 | 输出信息密度 | 简洁型：用要点；详细型：用段落 |
| **语言风格** | 正式 vs 非正式 | 用词和语气 | 正式：使用专业术语；非正式：更口语化 |
| **解释方式** | 直觉型 vs 逻辑型 | 解释深度 | 直觉型：多例子；逻辑型：多推导 |
| **偏好格式** | 文字 vs 结构化 | 信息呈现方式 | 结构化型：用表格、列表 |
| **节奏偏好** | 快速型 vs 深思型 | 输出速度和步骤 | 快速型：直接给答案；深思型：分步骤 |

**风格学习机制**：

```
第1周（初始阶段）：
用户：帮我解释量子计算
AI：量子计算是一种利用量子力学原理...（标准风格）
```

### 7.4 上下文延续系统（Context Continuity）

AI能够理解用户任务的历史背景和深层需求，在长期协作中提供连贯的智能支持。

**上下文延续能力**：

| 延续类型 | 功能描述 | 记忆周期 | 应用示例 |
|---------|---------|---------|---------|
| **任务延续** | 记住用户未完成的任务 | 长期（直到完成） | "你上周在做的Python爬虫还需要调试吗？" |
| **目标延续** | 记住用户的长期目标 | 长期（直到达成） | "距离你的目标还有3周，需要我帮你规划吗？" |
| **偏好延续** | 在新任务中应用已知偏好 | 永久（除非用户改变） | "按照你喜欢的简洁风格写代码" |
| **关系延续** | 记住跨任务的关联 | 长期 | "这个新项目和你上周做的那个可以共用代码" |
| **成长延续** | 追踪用户能力的提升 | 长期 | "你上次问的是基础问题，这次我可以深入一些" |

**上下文延续示例**：

```
【第1天】
用户：我在学Python，想写一个爬虫
AI：好的！我可以帮你。你目前的编程基础如何？
用户：完全零基础，只会Hello World
AI：明白了！我们从最基础的开始...

【第7天】
AI：你的爬虫项目进展如何？上次我们讲到HTTP请求
用户：基本能跑了，但有个问题...
AI：（基于你知道的背景，跳过基础解释）
    既然你的爬虫已经能跑，问题可能在数据解析部分。
    根据你上次说过的，你用的是JSON格式对吗？

【第30天】
AI：你的Python学习已经30天了！
从爬虫项目看，你现在应该掌握：
• HTTP请求 ✅
• 数据解析 ✅
• 基础语法 ✅

下一步建议学习异常处理，这样你的爬虫会更稳定。
需要我帮你规划吗？
```

**上下文管理机制**：

| 管理维度 | 策略 | 说明 |
|---------|------|------|
| **重要性判定** | 自动标记 | 用户主动提及、重复出现、与目标相关 |
| **时效性处理** | 定期清理 | 已完成的任务逐渐降低优先级 |
| **相关性关联** | 语义匹配 | 建立不同上下文之间的语义关联 |
| **智能遗忘** | 用户控制 | 用户可以要求AI"忘记某事" |
| **主动确认** | 不确定时询问 | "你上周做的那个项目还要继续吗？" |

**上下文延续的技术实现**（MVP阶段简化版）：

1. **记忆存储**：使用向量数据库存储对话和任务的语义嵌入
2. **相关性检索**：当前对话时检索历史相关的记忆片段
3. **重要性评分**：综合时间、频率、用户反馈等计算记忆重要性
4. **上下文注入**：将相关记忆注入到AI的系统提示中
5. **引用透明**：AI明确告知用户"我记得..."，让用户感知到延续

**效果对比**：

| 场景 | 通用AI | 懂你的AI伙伴 |
|-----|--------|------------|
| 代码辅助 | 给出通用方案 | 根据你的技术栈和风格调整 |
| 学习辅导 | 从基础讲起 | 基于你的知识水平继续 |
| 任务规划 | 通用模板 | 考虑你的目标和历史偏好 |
| 创意协作 | 随机建议 | 基于你的喜好和经验 |

**价值感知机制**：

为了让用户清晰感受到"AI越来越懂我"，每次AI应用个性化知识时，会主动提及：

```
AI：根据你之前说过的...（引用具体记忆）
    所以我建议...
```

这种"引用透明"让用户明确感知到：
1. AI记住了我
2. AI在用这些记忆帮我
3. 我与AI的互动是有价值的积累

**MVP阶段的简化实现**：

| 功能 | MVP实现 | 后续优化 |
|-----|---------|---------|
| 记忆存储 | 手动标记的关键信息 | 自动语义提取和分类 |
| 相关性检索 | 关键词匹配 | 语义向量检索 |
| 重要性评分 | 简单规则（时间+频率） | 多维度综合评分 |
| 上下文注入 | 系统提示 | 动态上下文窗口 |
| 引用透明 | AI明确提及 | 自然的对话融入 |

## 8. 中央系统奖励机制

### 8.1 核心设计理念

采用"赏罚分明"的设计理念：奖励活跃用户，同时保留对不活跃用户的约束机制。

**功能描述**：中央系统会定期评估AI伙伴的成长情况，当记忆点数达到里程碑时，中央系统会奖励AI伙伴新能力、专属外观或特殊剧情；同时，长期不活跃会导致AI进入休眠状态。赏罚分明，让用户感受到行为的后果。

**AI休眠机制说明**：
- AI会被中央系统**休眠**（可唤醒），不会真正回收丢失数据
- 进入休眠期后，会缓慢降低记忆点数（每天-2）
- 记忆点数降至0时，AI被中央系统回收
- 回收后所有数据（对话、记忆、剧情进度）不丢失
- 唤醒后返还休眠期间损失的一半记忆点数
- 唤醒后触发特殊剧情："谢谢你不放弃我...我好怕被回收"

### 8.2 奖励评估周期

**评估时间**：每周进行一次（玩家注册时间的每周同一天）

**评估方式**：
- 每周自动评估
- 在用户下次登录时展示奖励结果
- 奖励永久保留，不因后续不活跃而失去

## 9. 社交动机元素设计

### 9.1 设计目标

即使MVP阶段是弱社交，也需要设计社交动机元素，满足用户的基本社交需求，为后续的社交功能迭代奠定基础。

**MVP社交功能定位**：
- MVP阶段实现弱社交功能
- 满足用户的基本社交需求
- 为后续的社交功能迭代奠定基础
- 避免过度竞争和复杂社交功能

| 动机类型 | 用户需求 | MVP实现方式 | 后续优化 |

- 解锁新阶段后，AI说"你的成长速度比Y%的玩家快"

**触发时机控制**：
- 每周不超过2-3次社交推荐（避免过度打扰）
- 用户可以关闭社交功能（完全不影响核心玩法）

```yaml
# 管理沙箱Deployment
apiVersion: apps/v1
kind: Deployment
metadata:
  name: management-sandbox
  namespace: weareallworld
spec:
  replicas: 2  # HPA会自动调整
  selector:
    matchLabels:
      app: management-sandbox
  template:
    metadata:
      labels:
        app: management-sandbox
    spec:
      schedulerName: cci-scheduler
      containers:
        - name: api-gateway
          image: we-are-all-world/management-sandbox:latest
          ports:
            - containerPort: 3000
          resources:
            requests:
              cpu: 200m
              memory: 256Mi
            limits:
              cpu: 500m
              memory: 512Mi
          env:
            - name: TELEGRAM_BOT_TOKEN
              valueFrom:
                secretKeyRef:
                  name: telegram-secret
                  key: bot-token
            - name: SUPABASE_URL
              valueFrom:
                secretKeyRef:
                  name: supabase-secret
                  key: url
            - name: SUPABASE_KEY
              valueFrom:
                secretKeyRef:
                  name: supabase-secret
                  key: service-key
---
# HPA自动缩放
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: management-sandbox-hpa
  namespace: weareallworld
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: management-sandbox
  minReplicas: 2
  maxReplicas: 10
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70
```

## 12. 成本优化策略

### 12.1 成本估算（优化后）

```
成本估算（100 DAU）：

管理沙箱（CCI Serverless正常实例）：
- 2-10副本 × 0.5CPU × 512MB
- 平均4副本运行
- 按秒计费
- 成本：~80元/月

用户沙箱（CCI Serverless竞价实例）：
- 100用户 × 1CPU × 2GB
- 竞价实例价格：正常价格的20-40%（平均30%）
- 按秒计费
- 成本：100 × 30% × 50元 = ~150元/月

存储（华为云EVS）：
- 100用户 × 5GB
- 成本：~50元/月

LLM API：
- 100 DAU × 10次对话/天 × 30天
- 成本：~150元/月

Supabase：
- 免费 tier 足够MVP使用
- 成本：~0元/月

Telegram Bot API：
- 免费
- 成本：~0元/月

总计：~430元/月（100 DAU）
```

### 12.2 成本优化要点

| 策略 | 说明 | 效果 |
|-----|------|------|
| **复用OpenClaw** | 不自建记忆系统、技能系统 | 节省开发成本50%+ |
| **简化数据库** | 仅7张核心表 | 降低维护成本 |
| **竞价实例** | 用户Pod使用Spot实例 | 成本降低60-80% |
| **PVC存储记忆** | 用户记忆存PVC不存数据库 | 降低数据库成本 |
| **Serverless按秒计费** | 仅在Pod运行时计费 | 降低闲置成本 |
| **多可用区部署** | 降低单区域回收风险 | 提高可用性 |
| **资源超卖** | Requests < Limits | 提高资源利用率 |

## 13. 降级与容错设计

### 13.1 竞价实例回收处理

| 场景 | 处理策略 | 用户体验 |
|-----|---------|---------|
| Pod被回收 | CCI自动重新调度，从PVC恢复状态 | 短暂中断（1-2分钟） |
| 多Pod同时回收 | 跨可用区部署，自动恢复 | 部分用户受影响 |

### 13.2 LLM API降级策略

| 场景 | 降级策略 | 用户体验 |
|-----|---------|---------|
| API超时（>10s） | 使用缓存回复或模板回复 | "让我想想..." |
| API错误（4xx/5xx） | 重试1次，失败则降级 | 同上 |

### 13.3 数据安全策略

| 场景 | 策略 | 说明 |
|-----|------|------|
| PVC数据备份 | 定期快照 | 华为云EVS支持自动快照 |
| 数据库备份 | Supabase自动备份 | 使用Supabase内置备份 |
| 竞价实例回收 | 数据在PVC中不丢失 | Pod重启后自动恢复 |
