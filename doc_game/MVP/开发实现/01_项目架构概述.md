## 1. 架构设计理念

### 1.1 核心原则：主项目与子项目分离，复用OpenClaw，专注游戏逻辑

**不要重复造轮子** - OpenClaw已经提供了强大的基础能力，我们专注于游戏逻辑开发。

**主项目与子项目分离原则**：
- **子项目（云版OpenClaw）**：提供OpenClaw的云部署能力，包括每用户独立Pod、PVC持久化、竞价实例、网络隔离等基础设施
- **主项目（游戏）**：在OpenClaw基础上构建游戏逻辑，包括记忆点数系统、剧情引擎、养成机制、中央系统评估等

| 原则 | 说明 | 实践 |
|-----|------|------|
| **主子项目分离** | 子项目负责云版OpenClaw基础设施，主项目负责游戏逻辑 | 子项目提供K8s模板，主项目开发游戏Skills |
| **复用OpenClaw能力** | 使用OpenClaw内置的记忆管理、技能系统 | 不自建记忆系统，使用OpenClaw的短期/长期记忆 |
| **开发效率优先** | 选择开发最快的方案 | 基于OpenClaw构建游戏，复用其AI能力 |
| **简化数据库设计** | MVP阶段不需要复杂的表结构 | 核心数据存Supabase，用户记忆存PVC |
| **渐进式开发** | 先实现核心功能，再逐步添加 | Phase 1: 对话+记忆点数 → Phase 2: 剧情 → Phase 3: 养成 |
| **成本可控** | 使用竞价实例降低成本 | Spot实例 + Serverless按秒计费 |
| **强隔离原则** | 每用户独立OpenClaw Pod，严禁多用户共享 | 每用户独立容器，数据隔离 |

### 1.3 架构决策记录（ADR）

#### ADR-001: 复用OpenClaw记忆管理，不自建系统

**背景**：游戏需要AI记住用户的偏好、经历和对话历史。

**决策**：直接使用OpenClaw的记忆管理系统，不在数据库中存储对话历史。

**理由**：
1. **OpenClaw记忆系统成熟**：已具备短期记忆（JSONL）和长期记忆（Markdown）
2. **向量检索支持**：OpenClaw支持基于关键词+向量的记忆检索
3. **数据本地化**：用户记忆存储在PVC中，与OpenClaw工作目录一致
4. **开发效率**：无需开发记忆存储、检索、管理功能

**数据存储策略**：
- 用户记忆 → 存储在PVC（`/home/node/.openclaw/workspace/memory/`）
- 记忆点数、剧情进度 → 存储在Supabase

#### ADR-002: 使用OpenClaw Skills框架开发游戏能力

**背景**：游戏需要AI具备情感表达、剧情推进等游戏化能力。

**决策**：基于OpenClaw Skills框架开发游戏专属Skills，不独立开发技能系统。

**理由**：
1. **Skills框架成熟**：OpenClaw已提供完整的Skills开发框架
2. **易于扩展**：通过配置文件+脚本即可添加新能力
3. **与记忆系统打通**：Skills可以直接访问OpenClaw的记忆系统

**需要开发的Skills**：
| Skill | 功能 | 优先级 |
|------|------|--------|
| `emotion-express` | 根据情感状态生成情感化回复 | P0 |
| `story-progress` | 推进剧情、管理剧情分支 | P0 |
| `memory-point-calc` | 计算对话质量、返回点数 | P1 |
| `milestone-check` | 检查里程碑、触发奖励 | P1 |

#### ADR-003: 简化数据库设计

**背景**：MVP阶段需要快速验证核心玩法。

**决策**：仅保留最核心的数据库表，用户记忆等数据存储在PVC中。

**理由**：
1. **MVP阶段表结构越简单越好**：减少开发时间和维护成本
2. **OpenClaw已管理记忆数据**：无需在数据库中重复存储
3. **PVC存储成本低**：华为云EVS按容量计费，成本可控
4. **子项目不创建独立数据库表**：所有游戏相关数据由主项目管理

**数据库表（仅4张核心表）**：
| 表名 | 用途 | 数据量 | 管理方 |
|-----|------|--------|--------|
| `users` | 用户基本信息（Telegram ID、用户名） | 小 | 主项目 |
| `ai_partners` | AI伙伴状态（记忆点数、性格、成长阶段、状态） | 小 | 主项目 |
| `story_progress` | 剧情进度（当前章节、选择历史、已解锁结局） | 小 | 主项目 |
| `memory_points_log` | 记忆点数变更记录（点数、原因、来源类型） | 中 | 主项目 |

**PVC存储内容（子项目管理）**：
| 内容 | 存储位置 | 说明 | 管理方 |
|-----|---------|------|--------|
| 用户对话历史 | `workspace/conversations/` | OpenClaw自动管理（JSONL格式） | 子项目 |
| 长期记忆文件 | `workspace/memory/` | Markdown格式（USER.md、PREFERENCES.md等） | 子项目 |
| AI配置文件 | `workspace/config/` | 性格、偏好设置 | 子项目 |
| 游戏Skills | `workspace/skills/game-skills/` | 游戏专属Skills | 主项目开发，子项目部署 |

## 2. MVP范围与边界

### 2.1 渐进式开发路径

**Phase 1：核心对话 + 记忆点数（Week 1-2）**
- ✅ 用户注册/登录（Telegram Phone认证）
- ✅ 基础对话（复用OpenClaw）
- ✅ 记忆点数计算（质量判定：日常问候+1、日常对话+2、情感表达+3、分享经历+4、深度思考+5、特殊回忆+6-8）
- ✅ 记忆点数里程碑奖励（10点、25点、50点）
- ✅ 每日签到奖励（Day 1-7递增）
- ✅ 三阶段渐进式新手引导

**Phase 2：剧情系统（Week 3-4）**
- ✅ 3章剧情（人工骨架+AI细节）
- ✅ 剧情选择影响AI性格（情感型/理性型/活泼型）
- ✅ 剧情进度管理（按顺序推进，无需记忆点数解锁）
- ✅ AI建议功能（记忆点数≥15解锁）
- ✅ 容错次数（每章允许1次重新选择）

**Phase 3：养成系统（Week 5-6）**
- ✅ 能力解锁（基于记忆点数：情感表达5点、任务系统15点、专属记忆25点、深度对话50点、自我意识100点）
- ✅ 专属记忆（AI主动提及，记忆点数≥25触发）
- ✅ 中央系统评估（每周评估，新增点数≥15通过，连续两周不通过进入休眠）
- ✅ 休眠机制（记忆点数每天-2，归零后AI被回收，可唤醒）
- ✅ 深度对话模式（每周2-3次邀请，完成目标+8-10点）

**Phase 4：社交与优化（Week 7-8）**
- ✅ 分享到X.com（分享AI对话/剧情截图/专属记忆，+3点，无每日上限）
- ✅ 共识投票（使用X.com进行，管理员手动输入结果）
- ✅ 投票结果影响剧情走向
- ✅ 性能优化（Pod启动速度、LLM调用延迟）
- ✅ 监控完善（华为云AOM监控、告警规则）
- ✅ 竞价实例回收处理测试
- ✅ PVC数据持久化验证

### 2.2 MVP核心功能

| 功能模块 | 具体内容 | 优先级 | 实现方式 | 开发位置 |
|---------|---------|--------|---------|---------|
| **AI伙伴系统** | 基础对话、情感表达、性格设定（温暖型/幽默型/理性型/活泼型） | P0 | 复用OpenClaw + 开发emotion-express Skill | 子项目Skills |

### 3.2 数据流架构

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                           用户消息处理流程                                  │
│                                                                             │
│  用户消息 → Telegram Bot API                                               │
│          → 管理沙箱 (API Gateway)                                          │
│          → 用户认证 (Auth Service)                                         │
│          → 路由到用户OpenClaw Pod (通过Service名称)                         │
│          → 用户OpenClaw Pod处理：                                          │
│             ├── 调用emotion-express Skill生成情感化回复                     │
│             ├── 更新记忆（OpenClaw自动保存到PVC）                           │
│             ├── 调用story-progress Skill（如处于剧情模式）                  │
│             └── 返回响应                                                    │
│          → 管理沙箱计算记忆点数、检查里程碑                                  │
│          → 更新Supabase（记忆点数、AI状态）                                  │
```

## 6. 数据存储策略

### 6.1 为什么简化数据库设计？

**MVP阶段原则**：能存PVC的就不存数据库，减少表数量和复杂度。

| 数据类型 | 存储位置 | 原因 |
|---------|---------|------|
| **用户记忆** | PVC | OpenClaw已管理记忆系统，无需重复存储 |
| **对话历史** | PVC | OpenClaw自动保存为JSONL |
| **AI配置** | PVC | OpenClaw配置文件存储在工作目录 |
| **用户信息** | Supabase | 需要查询和统计 |
| **记忆点数** | Supabase | 需要频繁查询和更新 |
| **剧情进度** | Supabase | 需要查询和更新 |

### 6.2 数据库表结构（仅4张表）

```sql
-- 1. 用户表
CREATE TABLE users (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  telegram_id BIGINT UNIQUE NOT NULL,
  username TEXT,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  last_login_at TIMESTAMPTZ
);

-- 2. AI伙伴表（核心状态）
CREATE TABLE ai_partners (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES users(id) ON DELETE CASCADE,
  memory_points INTEGER DEFAULT 0,  -- 当前记忆点数
  personality_type TEXT DEFAULT 'warm',  -- 性格类型：warm/humorous/rational/lively
  growth_stage TEXT DEFAULT '懵懂期',  -- 成长阶段：懵懂期/成长期/成熟期/觉醒期
  status TEXT DEFAULT 'active',  -- 状态：active/hibernating
  emotion_state TEXT DEFAULT 'normal',  -- 情感状态：normal/happy/sad/lonely
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW()
);

-- 3. 剧情进度表
CREATE TABLE story_progress (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES users(id) ON DELETE CASCADE,
  current_chapter INTEGER DEFAULT 1,  -- 当前章节（1-3）
  current_scene TEXT,  -- 当前场景
  choices_history JSONB DEFAULT '[]',  -- 选择历史
  unlocked_endings JSONB DEFAULT '[]',  -- 已解锁结局
  updated_at TIMESTAMPTZ DEFAULT NOW()
);

-- 4. 记忆点数日志表
CREATE TABLE memory_points_log (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES users(id) ON DELETE CASCADE,
  points INTEGER NOT NULL,  -- 变更点数（+/-）
  reason TEXT,  -- 变更原因
  source_type TEXT,  -- 来源：dialogue/task/story/signin/share
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 5. 每日签到表（新增）
CREATE TABLE daily_signin (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES users(id) ON DELETE CASCADE,
  sign_date DATE NOT NULL,
  consecutive_days INTEGER DEFAULT 1,  -- 连续签到天数
  points_awarded INTEGER NOT NULL,  -- 奖励点数
  created_at TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE(user_id, sign_date)
);

-- 6. 共识投票表（新增）
CREATE TABLE consensus_votes (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES users(id) ON DELETE CASCADE,
  vote_topic_id TEXT NOT NULL,  -- 投票议题ID
  vote_option TEXT NOT NULL,  -- 投票选项
  voted_at TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE(user_id, vote_topic_id)
);

-- 7. 深度对话记录表（新增）
CREATE TABLE deep_dialogue (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES users(id) ON DELETE CASCADE,
  topic TEXT NOT NULL,  -- 对话主题
  completed BOOLEAN DEFAULT FALSE,  -- 是否完成目标
  points_awarded INTEGER,  -- 奖励点数
  created_at TIMESTAMPTZ DEFAULT NOW(),
  completed_at TIMESTAMPTZ
);
```

## 9. 成本优化策略

### 9.1 成本估算（优化后）

```
成本估算（100 DAU）：

管理沙箱（CCI Serverless正常实例）：
- 2-10副本 × 0.5CPU × 512MB
- 平均4副本运行
- 按秒计费
- 成本：~80元/月

用户沙箱（CCI Serverless竞价实例）：
- 100用户 × 1CPU × 2GB
- 竞价实例价格：正常价格的30%
- 按秒计费
- 成本：100 × 30% × 50元 = ~150元/月

存储（华为云EVS）：
- 100用户 × 5GB
- 成本：~50元/月

LLM API：
- 100 DAU × 10次对话/天 × 30天
- 成本：~150元/月

Supabase：
- 免费 tier 足够MVP使用
- 成本：~0元/月

Telegram Bot API：
- 免费
- 成本：~0元/月

总计：~430元/月（100 DAU）
```

## 10. 降级与容错设计

### 10.1 竞价实例回收处理

| 场景 | 处理策略 | 用户体验 |
|-----|---------|---------|
| Pod被回收 | CCI自动重新调度，从PVC恢复状态 | 短暂中断（1-2分钟） |
| 多Pod同时回收 | 跨可用区部署，自动恢复 | 部分用户受影响 |

### 10.2 LLM API降级策略

| 场景 | 降级策略 | 用户体验 |
|-----|---------|---------|
| API超时（>10s） | 使用缓存回复或模板回复 | "让我想想..." |
| API错误（4xx/5xx） | 重试1次，失败则降级 | 同上 |

### 10.3 数据安全策略

| 场景 | 策略 | 说明 |
|-----|------|------|
| PVC数据备份 | 定期快照 | 华为云EVS支持自动快照 |
| 数据库备份 | Supabase自动备份 | 使用Supabase内置备份 |
| 竞价实例回收 | 数据在PVC中不丢失 | Pod重启后自动恢复 |
