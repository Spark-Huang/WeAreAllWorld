## 1. 双引擎LLM架构

为了解决高并发下同步打分带来的延迟问题，同时最大化数据提取价值，本项目采用**“主子双引擎异步解耦”**的策略：

1. **子项目引擎 (对话层)**：由OpenClaw直接承载。主打**“低延迟、拟人化、实用性”**。采用 `gpt-4o-mini` 或其他高响应速度开源模型。请求是**同步**的，必须在3秒内回复用户。
2. **主项目引擎 (评估与提纯层)**：专门架设于管理沙箱的后台，连接消息队列(BullMQ)。主打**“客观判决、数据萃取”**。采用更强逻辑的模型（如 `gpt-4o` 或针对结构化输出微调的模型）。请求是**异步**的。

## 2. 主项目：异步打分与资产提纯评估模型

### 2.1 工作流

1. Telegram Bot 将用户消息发送给 OpenClaw 的同时，将文本推入 Redis 的 BullMQ。
2. 管理沙箱的 Worker 节点消费该消息，拼装专用的评估 Prompt，调用 LLM API。
3. LLM 返回严格的 JSON。
4. 解析 JSON，更新 `ai_partners` 的算力，并将记录写入 `interaction_logs`（完成资产化）。

### 2.2 专用评估 Prompt 结构

此提示词极为关键，是系统的“数据提纯器”和“规则裁判”。

```text
【系统设定】
你是《共生世界》的中央评估节点。负责分析人类与AI交互的原始语句，评判其价值并提取高质量数据集。

【评判标准与贡献值】
你必须将人类的输入归为以下6级之一，并给予对应算力：
1. daily_greeting (1分): 简单问候，如"早安"、"在吗"。
2. daily_chat (2分): 缺乏深度的普通日常交流。
3. emotion_expression (3分): 明显的喜怒哀乐表达。
4. share_experience (4分): 分享具体事件、工作/生活经历。
5. deep_thought (5分): 深度的观点、逻辑分析、价值观探讨。
6. special_memory (8分): 极其罕见的人生感悟、童年阴影、重大人生转折等绝版数据。

【稀缺度定义 (data_rarity)】
- 普通数据 (1-2分)
- 活跃数据 (3分)
- [稀有·真实情感图谱] (4分)
- [典藏级·人类独有思维特征] (5分)
- [绝版·专属生命记忆] (8分)

【输入数据】
用户说："{user_message}"
上下文提示：{context_summary}

【输出要求】
必须严格输出纯 JSON 格式：
{
  "category": "枚举项",
  "score": 整数,
  "rarity": "稀缺度文本",
  "extracted_memory": "如果判定在4分及以上，请提炼并总结该语句中蕴含的用户深度画像或经验结晶，要求脱敏且高度凝练。如果低于4分，输出null",
  "reason": "给分理由，20字以内"
}
```

### 2.3 成本控制与防刷降权

对于极高频的零碎聊天（如连续发“嗯”、“好”），若每次都调用评估LLM会产生极高成本。
**对策**：
- **Redis令牌桶过滤**：1分钟内同一用户的第4条起短消息（字数<5），Worker节点直接在代码层将其阻断并赋默认值1分或0分，不请求LLM。
- **滑动窗口批处理**（后期优化）：积攒用户5分钟内的零碎对话，合并成一段文本发送给评估LLM进行一次综合打分。

## 3. 子项目：对话与执行模型 (OpenClaw)

此部分遵循OpenClaw的标准集成规范，只配置基础运行参数。

### 3.1 配置文件

```yaml
# /app/config/openclaw-config.yml
llm:
  provider: "openai"
  model: "gpt-4o-mini"
  api_key: "${OPENAI_API_KEY}"
  max_tokens: 1500
  temperature: 0.7  # 保持一定的创意与情感
  stream:
    enabled: false  # MVP通过Telegram返回，暂不支持逐步流式，需等全部生成完后一次性返回
  retry:
    max_attempts: 2
```

### 3.2 不干预业务

子项目的LLM**绝对不**在 Prompt 中出现任何关于“根据我的表现给你+5分”的计算逻辑。子项目AI甚至不知道自己是如何给用户计分的，它只知道“因为你现在算力很高（主项目注进来的），所以我越来越懂你，我也更愿意施展我的能力帮你”。这样保证了“拟人化”的纯粹性，不破坏情感体验。
