## 1. 架构设计理念

### 1.1 核心原则：主子项目分离，坚决不重复造轮子

天下一家（WeAreAll.World）并非又一个普通的AI套壳应用，而是一个**基础设施**。我们通过"生存压力+情感飞轮"来源源不断地收集高质量的私有数据。

**架构原则**：
- **子项目（云版OpenClaw基础设施）**：负责底层通用能力（LLM调用、记忆持久化、工具链执行）。每个用户拥有独立的Pod沙箱与PVC存储，保证数据隔离。**我们绝不自己开发基础的大模型对话和记忆引擎。**
- **主项目（共生规则系统引擎）**：负责所有的业务规则、质量判定、数据提纯、资产化量化与生命周期管理。**这是本产品最核心的护城河。**

### 1.2 核心业务飞轮

**情感连接 × 实用能力 = 极高迁移成本**
1. **获取贡献值**：用户与AI交互，主项目（贡献值判定引擎）异步评估数据质量并发放算力。
2. **避免休眠回收**：中央评估引擎每周核算，不达标则休眠。赋予AI真正的"生存压力"。
3. **解锁特权能力**：算力里程碑晋升后，热更新子项目权限，AI获得更强的实用能力（OpenClaw高级Skills）。
4. **效能反哺情感**：AI越懂用户，工作效能越高，用户越离不开，从而持续输入更多高质量隐私数据。

## 2. MVP范围与边界

### 2.1 MVP只验证极简且致命的飞轮

为了实现极简MVP与低风险高赔率的落地策略，MVP阶段**坚决砍掉**重度游戏化（3D、复杂商城、多线分支文字引擎等），专注验证"高质量对话 -> 累积稀缺数据 -> 避免AI死亡 -> 解锁超强个性化实用工具"。

### 2.2 MVP核心功能

| 功能模块 | 职责与实现方式 | 归属 |
|---------|---------|---------|
| **AI底座与记忆** | 基础对话、情感表达、记忆持久化（复用云版OpenClaw） | 子项目 |
| **贡献值判定** | 拦截消息，通过BullMQ队列异步调用专用评估LLM，对消息打分(1-8分)及提纯资产 | 主项目 |
| **中央评估与休眠** | 每周核算（≥15点/周），连续不达标则进入休眠阻断状态，每日扣除算力 | 主项目 |
| **能力映射升级** | 监听算力里程碑，通过 `X-User-Context` 向子项目动态注入高级权限和知识库权重 | 主项目 |
| **轻量级引导与剧情** | 人工骨架+AI细节，固定章节的对话式推进 | 主项目/子项目协同 |
| **Telegram Bot接入** | Webhook路由，状态拦截与透传 | 主项目 |

## 3. 系统总体架构

系统采用严格的"主子项目分离"和"异步解耦"架构。

```mermaid
graph TD
    A[Telegram Bot Webhook] --> B(核心路由分发器 API Gateway)

    subgraph 主项目: 共生规则系统 (SaaS Control Plane)
        B --> C{核心业务调度器}

        C -.->|异步消息| MQ[(Redis / BullMQ)]
        MQ --> D[贡献值判定引擎]
        D --> D1[专用评估LLM打分]
        D --> D2[数据脱敏与提纯入库]

        C --> E[中央评估与生命周期引擎]
        E --> E1[周常Cron核算 >=15点]
        E --> E2[衰减与休眠阻断]

        C --> F[里程碑与能力映射引擎]
        F --> F1[阶段晋升事件]
    end

    subgraph 子项目: 云版OpenClaw (User Sandbox Infrastructure)
        C -.-> |Context Injection / 透传聊天| I[用户专属 OpenClaw Pod 集群]
        I -.-> |Tools/Skills Exec| J[基础大模型 & 长期/短期记忆 PVC]
    end

    subgraph 核心数据层
        D -.-> H[(Supabase PostgreSQL)]
        E -.-> H
        F -.-> H
        C -.-> H
    end
```

## 4. 核心模块与职责分工

### 4.1 管理沙箱（主项目核心）

管理沙箱是无状态的Node.js服务（可水平扩展），处理一切与"规则和资产"相关的工作。

1. **核心路由（Bot Webhook）**：
   - 鉴权、用户映射。
   - 检查AI是否休眠：若休眠，拦截并返回警告；若活跃，透传请求给对应的OpenClaw Pod。
2. **贡献值判定引擎（异步旁路）**：
   - 聊天请求透传给OpenClaw以保证极速响应。同时将用户输入推入BullMQ。
   - 后台Worker调用专用评估LLM提取数据价值，生成 JSON 包含 `score`, `category`, `rarity`, `extracted_memory`。
3. **生命周期引擎（Cron定时任务）**：
   - 周常核算（`weekly_evaluation`）：扫描新增点数，低于15点增加违规计数，达到2次触发休眠。
   - 每日衰减（`daily_decay`）：对休眠状态AI每日扣除2点。
4. **里程碑映射引擎**：
   - 监听总点数，升级阶段（懵懂期 -> 成长期 -> 成熟期 -> 觉醒期）。

### 4.2 用户沙箱（云版OpenClaw子项目）

为每个用户分配一个轻量级的CCI Serverless竞价实例（Spot）。

1. **OpenClaw Engine**：加载用户的独立环境。
2. **持久化PVC存储**：保存JSONL对话记录、Markdown长期记忆、个性化配置。
3. **特定游戏技能（Skills）**：
   - `emotion-express`：根据主项目传来的上下文调整回复情绪。
   - `story-progress`：执行简单剧情内容。

**重要变更**：在最新架构中，**质量判定（点数计算）不再由子项目的Skill执行**，而是完全收拢到主项目的【贡献值判定引擎】中通过旁路异步LLM执行，以保证计分权威性和架构清晰。

## 5. 主项目与子项目协同

### 5.1 上下文注入 (Context Injection)

主项目在每次转发聊天请求给子项目时，无需子项目反向查库，而是通过 HTTP Header `X-User-Context` 携带最新的生命周期状态。

```typescript
interface UserContext {
  userId: string;
  currentSurvivalPower: number;   // 当前贡献值
  totalSurvivalPower: number;     // 累计贡献值（用于判定里程碑）
  personalityType: string;        // AI性格
  growthStage: string;            // 成长阶段
  status: string;                 // active / hibernated
  practicalAbilityBoost: number;  // 实用能力增强系数(由里程碑决定)
}
```

### 5.2 同步与异步解耦的工作流

```
用户消息 -> Telegram Webhook -> 路由分发器
  ├─(1. 阻断检查)─> 若AI处于休眠，直接返回休眠提示，流程结束。
  ├─(2. 同步透传)─> 携带 X-User-Context 发送至 user-svc-{uid}:18789/api/chat
  │                 └─> OpenClaw 处理记忆、调用底层大模型 -> 返回回复 -> 路由分发器返回给用户 (保证<3秒响应)
  └─(3. 异步提纯)─> 将消息推入 BullMQ
                    └─> Worker 消费 -> 调用专属评估LLM -> 生成分数与高纯度数据 -> 写入 interaction_logs -> 更新 ai_partners 算力
```

## 6. 数据存储与隔离策略

核心原则：**状态分离**。业务流转数据存入Supabase，AI个人对话记忆存入隔离PVC。

### 6.1 Supabase (主项目掌控)

仅包含核心资产与状态控制表：
- `users`: 用户基础映射。
- `ai_partners`: 算力账本与生命状态表（核心中枢）。
- `interaction_logs`: 交互质量与数据资产池（防刷与未来数据交易凭证）。
- `central_evaluations`: 中央评估审计流水（生死判决记录）。
- `story_progress`: 剧情进度。

### 6.2 云服务 EVS / PVC (子项目专属)

每个用户的独立沙箱挂载私有PVC，存储非结构化记忆：
- `/conversations/*.jsonl`
- `/memory/*.md` (长期图谱库)

这种架构使得子项目Pod即使因为竞价实例被回收而终止，数据依然安全，用户唤醒时可瞬间重新挂载恢复状态。

## 7. 成本与性能优化

1. **异步LLM评估机制**：
   - 将影响用户体感的聊天生成（同步）与极其消耗资源的打分/信息提纯（异步）分离。聊天快速返回，打分在后台按需削峰执行。
2. **云版OpenClaw极致降本**：
   - 采用云服务 CCI 竞价实例（Spot），成本仅为常规实例的20%-30%。
   - Serverless 按秒计费，用户不活跃时可自动释放计算资源，仅保留极低成本的 EVS 存储。
3. **数据提纯复用**：
   - 在打分环节（异步），评估LLM顺便抽取出"高纯度差异化知识"，一次LLM调用完成两项任务（质量定级+数据资产化），避免冗余分析。
