## 2. 环境与基建申请

### 2.1 主项目 (管理沙箱)
- **运行环境**：Node.js >= 20, 云服务 CCE (普通节点，常驻高可用 2 副本)。
- **中间件**：Redis 实例 (用于 BullMQ 和频率限制，内存 >= 2G 即可)。
- **BaaS**：Supabase 托管版本 (MVP 阶段免费 Tier，支持 PostgreSQL 15+)。
- **LLM Token池**：优先使用 `gpt-4o-mini`。预估开发+联调期 20美元额度。

### 2.2 子项目 (用户沙箱)
- **运行环境**：云服务 CCI Serverless 竞价实例 (Spot)。无基建成本，拉起一个计费一次。
- **存储**：云服务 EVS 云硬盘 (每用户 5G，按挂载量计费)。

## 3. 部署架构与容灾策略

### 3.1 Serverless 资源释放与重挂载 (最关键)

为了省钱，子项目(OpenClaw) 的 Pod 生命周期无需与玩家状态强绑定。
- **长时间无通信释放**：如果 K8s 监测到某个 `user-svc-uid` 超过 2小时无请求，可以主动由管理面发出指令销毁该 Pod (缩容为0)。
- **唤醒重挂**：由于对话历史(`.jsonl`) 和长期画像(`.md`) 统统存放在用户的 EVS PVC 中，当用户 2 小时后再次发送消息，管理面发现 Pod 消失，会阻塞该请求3秒并立刻调起 K8s Deployment。新 Pod 挂载原 PVC 后，立刻恢复上一次聊天的记忆。对用户而言只有极其短暂的首包延迟。

### 3.2 MQ 堆积容灾
如果评估 LLM 遭遇大面积降级或拥堵，导致 BullMQ 中待打分消息堆积十万条。
- **对策**：不影响前台聊天。用户依然能立刻得到 OpenClaw 的回复。只要队列没有满，系统将在闲时缓慢消化打分请求。因为结算机制是以“周”为单位的，短暂的积分延迟入账不会导致业务系统崩溃。
